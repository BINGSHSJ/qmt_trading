# AGENTS.md｜本地量化控制台 Codex 长期执行规则

> 本文件是 Codex 每次进入仓库都必须优先读取的长期工程约束。  
> 详细需求见 `docs/`，每次开发只能根据当前阶段计划和相关模块文档执行，不允许一次性自由发挥开发完整项目。

## 1. 项目定位

项目名称：本地量化控制台 / Local Quant Console。  
当前用途：本地个人使用。  
未来目标：模块命名、目录结构、数据库结构支持商业化扩展。  
第一版核心链路：

```text
系统启动 → 环境检测 → 数据同步 → Python 策略 → 回测研究 → 交易执行 → 总览展示
```

## 2. 固定主菜单

主菜单统一四个汉字，不得擅自改名：

```text
总览看板
数据中心
策略开发
回测研究
交易执行
系统管理
```

## 3. 技术栈固定

前端：React + TypeScript + Vite。  
UI：Ant Design + ProComponents。  
图表：ECharts + Lightweight Charts。  
代码编辑器：Monaco Editor。  
后端：Python + FastAPI。  
数据库：SQLite。  
QMT 对接：通过独立 Adapter 封装 xtquant / MiniQMT。  
启动方式：本地一键启动脚本。

第一版不得默认引入 Redis、Celery、Docker、Kubernetes、微服务、云部署。

### 3.1 当前批准的轻量前端实现偏差

截至 2026-05-10，当前项目未安装 `@ant-design/pro-components` 和 `echarts`。这是为遵守“第一版不加重依赖、不引入新 UI 框架、不做大重构”的边界而批准的轻量实现偏差，不视为必须立即补齐的问题。

当前第一版执行口径：

- 表格继续使用 Ant Design Table 内核 + 自研 `DataTable` 轻量封装。
- 卡片、指标、状态、日志、危险确认等继续使用自研轻量组件，如 `SectionCard`、`MetricCard`、`StatusChip`、`LogDrawer`、`DangerActionMenu`。
- K 线和回测图表优先沿用当前已跑通实现；不要仅为了文档中的 ECharts/ProComponents 字样而新增依赖。
- 后续只有在明确阶段计划要求、且不破坏主链路时，才评估是否引入 ProComponents 或 ECharts。

### 3.2 当前真实 QMT 运行口径

截至 2026-05-11，系统业务运行口径调整为“真实 QMT 数据优先、业务 Mock 下线”：

- 默认配置必须使用真实 QMT 模式，`simulation_mode=false`。
- 数据中心、总览看板、系统管理和交易执行的日常页面不得把 Mock 作为普通用户可选入口。
- 真实 QMT 不可用时，页面必须给出中文错误、技术详情和下一步建议；不得自动切回 Mock 生成业务数据。
- Mock / Fake / Test Adapter 只允许用于自动化测试、离线回归和开发排障，必须与真实业务数据隔离，不得混入默认业务视图。
- 回测仍然只读取已经落 SQLite 的历史行情数据，不调用真实 QMT 下单接口。
- 真实下单仍必须经过交易执行模块、人工确认和幂等保护；不得默认开启自动实盘交易。

## 4. 明确不做

第一版禁止实现以下内容：

- 多用户系统
- 登录权限
- 角色管理
- 多租户
- 审批流
- 复杂风控中心
- 商业化报表中心
- AI 工作台
- 消息中心
- 条件筛选建策略
- 拖拽策略
- 无代码策略生成器
- 参数网格寻优
- 多策略组合
- 默认自动实盘交易

## 5. 策略规则

策略只支持 Python 新建、导入、编辑和运行。  
不做条件筛选器，不做拖拽策略，不做无代码生成策略。

Python 策略必须遵守统一接口：

```python
class Strategy:
    name = "策略名称"
    version = "1.0.0"
    description = "策略说明"
    params = {}

    def __init__(self, context):
        self.context = context

    def run(self):
        signals = []
        return signals
```

策略只负责生成信号，不允许直接调用 QMT 下单。  
策略不得直接拿到 QMT 原始对象、交易服务对象、数据库连接。  
策略只能通过 `StrategyContext` 读取受控数据。  
Codex 不得覆盖 `strategies/user/` 目录中的用户策略文件。

## 6. 交易规则

所有下单必须经过交易执行模块。  
交易模块必须记录 `local_order_id`、`qmt_order_id`、订单来源、信号 ID、策略名称、订单状态。  
信号下单必须防重复提交。  
按钮提交后必须锁定，接口必须幂等。  
下单前确认弹窗默认开启。  
第一版只保留最小保护，不做复杂风控中心。

## 7. 数据规则

所有数据先落 SQLite，再给前端、策略、回测和交易使用。  
QMT 只是第一版数据源，架构不得写死为只能支持 QMT。  
列表接口必须分页、筛选、排序。  
禁止前端一次性加载全量数据。  
分钟 K 第一版不得默认全市场多年全量同步，必须按需同步、增量同步。

## 8. 长任务规则

以下操作必须作为长任务执行并返回 `task_id`：

- 数据同步
- 策略运行
- 回测任务
- 环境检测
- 备份恢复
- 大批量导出

前端通过任务接口轮询进度。  
长任务必须支持状态、进度、日志、开始时间、结束时间、错误信息。

## 9. UI/UX 规则

所有核心功能必须可视化操作。  
用户不需要手动运行 Python 脚本。  
用户不需要打开 SQLite 数据库。  
用户不需要手动修改 JSON 配置。  
所有错误必须有中文解释，同时保留技术详情。  
所有页面必须有加载态、空状态、错误态、刷新按钮、最近更新时间。

## 10. 后端规则

后端必须分层：API / Schema / Service / Repository / Adapter / Task / Core。  
API 层不得直接访问数据库。  
业务逻辑不得写在 API 路由里。  
QMT 调用只能出现在 Adapter 层。  
所有写入操作必须记录操作日志。  
所有接口必须返回统一响应结构。

## 11. SQLite 规则

启用 WAL。  
批量写入必须使用事务。  
禁止逐条 commit。  
关键表必须有唯一索引和查询索引。  
必须维护同步游标 `sync_cursor`。  
日志必须支持保留天数、归档、清理。

## 12. 测试规则

每次开发后必须说明测试结果。  
后端至少运行 pytest。  
前端至少运行 lint / typecheck。  
涉及页面流程时，优先补充 Playwright 测试。  
涉及 QMT 时必须提供测试隔离替身，不能依赖真实 QMT 才能跑自动化测试；但业务运行模式不得默认使用 Mock。

## 13. 每次任务输出要求

Codex 每次完成任务必须输出：

1. 本次实现了什么。
2. 修改了哪些文件。
3. 新增了哪些接口。
4. 新增或修改了哪些数据库表。
5. 如何启动。
6. 如何测试。
7. 哪些功能是真实实现，哪些仅为测试隔离替身。
8. 是否触碰了禁止范围。
9. 后续建议的下一步任务。

## 14. 变更规则

不得新增未定义主模块。  
不得擅自改变六大菜单名称。  
不得把后续商业化能力提前做重。  
发现需求冲突时，先停止实现并输出冲突说明和建议方案。  
当前阶段只实现当前计划文件定义的范围。
