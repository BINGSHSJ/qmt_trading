# 00_项目总览

> 本文件用于让 Codex 理解项目定位、主链路、模块边界、技术方向和第一版冻结范围。详细实现规范见其他独立文档。

## 0. 版本总说明

本版本是在 v1.1 的基础上继续收敛需求，重点解决以下问题：

1. 防止后续越改越乱。
2. 明确每个模块的功能边界。
3. 明确前端、后端、数据库、任务、性能、UI/UX 的实现约束。
4. 明确策略只使用 Python 新建策略，不做筛选器、拖拽器、条件组合器。
5. 明确第一版是轻量 MVP，但架构命名和模块边界按未来可商业化产品设计。
6. 明确所有核心操作都必须在可视化后台完成，用户不需要打开命令行、数据库或配置文件。

本文件是项目第一版开发的**主规格书**。Codex 开发时必须以本文件为准，不得自行扩展重功能。

## 1. 给 Codex 的总指令

### 1.1 必须遵守

1. 所有核心功能必须通过后台页面操作。
2. 用户不需要手动运行 Python 脚本。
3. 用户不需要打开 SQLite 数据库。
4. 用户不需要手动修改 JSON 配置文件。
5. 用户不需要看英文报错。
6. 所有错误必须有中文解释，同时保留技术详情，方便复制给 AI 排查。
7. 主菜单和二级 Tab 菜单统一使用 **四个汉字**。
8. 策略只支持 **Python 新建策略**。
9. 不做条件筛选建策略。
10. 不做拖拽策略。
11. 不做无代码策略生成器。
12. QMT 只是第一版数据源，架构上不要写死成只能支持 QMT。
13. 所有数据先落本地 SQLite，再给前端、策略、回测和交易使用。
14. Python 策略只生成信号，不直接调用 QMT 下单。
15. 下单统一经过交易执行模块。
16. Codex 不得随意覆盖 `strategies/user/` 目录中的用户策略文件。
17. 所有长任务必须返回 `task_id`，前端通过任务状态查询进度。
18. 所有列表接口必须分页、筛选、排序，禁止一次性返回全量数据。
19. 所有页面必须有加载态、空状态、错误态、刷新按钮、最近更新时间。
20. 第一版不追求功能多，追求流程完整、稳定、清晰、可维护。

### 1.2 明确不做

| 不做内容 | 原因 |
|---|---|
| 多用户系统 | 当前本地个人使用 |
| 登录权限 | 第一版避免复杂化 |
| 角色管理 | 第一版无必要 |
| 多租户 | 商业化后再做 |
| 复杂风控中心 | 用户明确不需要重风控 |
| 审批流 | 个人使用无意义 |
| 条件筛选建策略 | 已取消，统一 Python 策略 |
| 拖拽策略 | 开发成本高，且不适合真实策略 |
| AI 工作台 | 暂不需要 |
| 消息中心 | 暂不需要 |
| 商业化报表 | 后续再做 |
| 参数网格寻优 | 第一版不做 |
| 多策略组合 | 第一版不做 |
| Redis | 第一版不引入 |
| Celery | 第一版不引入 |
| Docker | 第一版不作为默认启动方式 |
| 微服务 | 第一版必须单体架构 |
| Kubernetes | 完全不需要 |
| 云端部署 | 第一版本地运行 |

### 1.3 第一版必须完整跑通的主链路

```text
启动系统
  ↓
系统管理完成 QMT 环境检测
  ↓
数据中心同步 QMT 账户、持仓、委托、成交、行情
  ↓
策略开发中新建或导入 Python 策略
  ↓
策略运行产生信号
  ↓
回测研究使用同一策略进行回测
  ↓
交易执行支持手动下单和信号下单
  ↓
同步委托成交结果
  ↓
总览看板展示状态、资产、信号、交易、任务
```

## 2. 产品定位

### 2.1 产品名称

中文名称：**本地量化控制台**  
英文代号：`Local Quant Console`

### 2.2 产品定位

本项目是一个本地运行的 A 股量化操作系统，当前只给个人使用，但模块命名、代码结构和数据结构要能支持未来商业化扩展。

它不是行情终端，也不是大型多用户量化平台。第一版只围绕四件事：

```text
数据同步 → 策略开发 → 回测研究 → 交易执行
```

### 2.3 用户画像

| 项目 | 描述 |
|---|---|
| 当前用户 | 个人投资者，主要使用 QMT/MiniQMT |
| 代码能力 | 不懂代码，但可以让 AI 写 Python 策略 |
| 操作习惯 | 希望全部在可视化后台完成 |
| 策略方式 | Python 策略文件，由 AI 辅助编写 |
| 数据来源 | 第一版以 QMT/MiniQMT 为主 |
| 部署方式 | 本地电脑运行 |
| 未来方向 | 未来可能商业化，扩展多数据源、多账户、多策略 |

### 2.4 核心设计原则

| 原则 | 说明 |
|---|---|
| 化繁为简 | 只保留当前确实需要的功能 |
| 名字正规 | 模块命名按产品化系统设计 |
| 功能轻量 | 不做企业级重系统 |
| 页面优先 | 不懂代码也能操作 |
| 策略代码化 | 策略统一使用 Python 文件 |
| 数据先落库 | 所有数据统一入库再使用 |
| 长任务可见 | 同步、策略、回测、备份都要有进度 |
| 交易可追踪 | 下单、撤单、成交必须有记录 |
| 错误可排查 | 中文说明 + 技术详情 + 复制给 AI |
| 性能可控 | 避免全量加载、阻塞页面、无索引查询 |
| 未来可扩展 | 当前轻量，但边界清晰 |

## 4. 总体架构设计

### 4.1 架构总览

第一版采用本地单体架构，不做微服务，不引入 Redis/Celery/Docker 作为默认依赖。

```text
React + TypeScript 前端
        ↓ HTTP API
FastAPI 后端
        ↓
Service 服务层
        ↓
Repository 数据访问层
        ↓
SQLite 本地数据库
        ↓
QMT / MiniQMT / xtquant 适配层
```

### 4.2 前端架构

前端采用页面模块化架构。

```text
frontend/src/
├── app/                 # 路由、布局、全局入口
├── pages/               # 6 个主菜单页面
├── components/          # 通用组件
├── services/            # API 请求封装
├── stores/              # 轻量全局状态
├── hooks/               # 通用 hooks
├── utils/               # 工具函数
├── types/               # TypeScript 类型
└── theme/               # UI 主题变量
```

前端必须遵守：

1. 页面通过路由懒加载。
2. Monaco Editor、ECharts、Lightweight Charts 必须懒加载。
3. 所有接口统一经过 `services/request.ts`。
4. 所有接口错误统一转换为中文错误提示。
5. 表格统一使用分页，不允许一次加载全量。
6. 长任务统一显示进度条、状态、日志入口。
7. 所有危险操作使用确认弹窗。
8. 页面刷新不能丢失当前 Tab、筛选条件和任务状态。
9. 任何页面都不能依赖浏览器控制台排查错误。

### 4.3 后端架构

后端采用 API → Service → Repository → Adapter 分层。

```text
backend/
├── api/                 # FastAPI 路由层，只负责参数校验和响应
├── schemas/             # Pydantic 请求/响应模型
├── services/            # 业务服务层
├── repositories/        # 数据库访问层
├── adapters/            # QMT 等外部适配层
├── tasks/               # 长任务执行器
├── core/                # 配置、日志、异常、依赖注入
├── models/              # ORM 或 SQLModel 模型
└── utils/               # 工具函数
```

后端必须遵守：

1. API 层不得写业务逻辑。
2. Service 层不得直接拼 SQL，统一走 Repository。
3. QMT 调用只能出现在 Adapter 层。
4. 策略代码不能拿到 QMT 原始对象。
5. 交易下单必须经过 Trading Service。
6. 所有异常必须转成统一错误结构。
7. 所有写操作必须记录 `operation_log`。
8. 所有长任务必须写入 `runtime_task`。
9. 所有任务日志必须写入对应日志表或统一日志文件。
10. 所有数据库写入必须使用事务。

### 4.4 数据架构

核心原则：**数据中心是数据入口，SQLite 是单一事实来源。**

```text
QMT 原始数据
  ↓
Adapter 标准化
  ↓
Repository 批量 upsert
  ↓
SQLite 标准表
  ↓
前端 / 策略 / 回测 / 交易读取
```

禁止：

1. 前端直接调用 QMT。
2. 策略直接调用 QMT。
3. 回测直接调用 QMT。
4. 交易状态绕过数据库只看接口返回。
5. 同一份业务数据多处重复维护。

### 4.5 任务架构

同步、策略运行、回测、环境检测、备份恢复都属于长任务。

统一任务流程：

```text
前端点击按钮
  ↓
POST 创建任务
  ↓
后端立即返回 task_id
  ↓
后台线程/任务执行器执行
  ↓
runtime_task 更新进度
  ↓
前端轮询任务状态
  ↓
任务完成后跳转结果或刷新页面
```

任务必须支持：

| 能力 | 要求 |
|---|---|
| 任务状态 | pending / running / success / failed / cancelled |
| 任务进度 | 0-100，可估算也可以阶段式 |
| 任务日志 | 可实时追加和查看 |
| 任务取消 | 策略运行、回测、长同步必须可取消 |
| 失败原因 | 中文原因 + 技术详情 |
| 重试能力 | 同步、环境检测、备份可重试 |

### 4.6 模块边界

| 模块 | 负责 | 不负责 |
|---|---|---|
| 总览看板 | 展示摘要、快捷入口 | 不承载复杂配置 |
| 数据中心 | 数据源、同步、质量、字典 | 不做下单 |
| 策略开发 | Python 策略文件、运行、信号 | 不直接下单 |
| 回测研究 | 使用策略和历史数据做验证 | 不调用 QMT 实盘交易 |
| 交易执行 | 手动/信号下单、撤单、状态同步 | 不写策略逻辑 |
| 系统管理 | 设置、检测、日志、备份 | 不做业务交易决策 |

## 5. 技术选型

### 5.1 总体选型

| 层级 | 技术 | 说明 |
|---|---|---|
| 前端框架 | React + TypeScript | 类型清晰，适合中后台 |
| 构建工具 | Vite | 本地开发快，配置轻 |
| UI 组件 | Ant Design | 中后台组件完整 |
| 高级表格 | ProTable / Ant Design Table | 数据表格多，适合后台 |
| 图表 | ECharts | 绩效、任务、统计图 |
| K 线图 | Lightweight Charts | 金融 K 线展示 |
| 代码编辑器 | Monaco Editor | Python 策略编辑 |
| 后端框架 | FastAPI | Python API 开发效率高 |
| 数据库 | SQLite | 本地个人版足够，低复杂度 |
| 数据处理 | pandas / numpy | 策略和回测计算 |
| QMT 对接 | xtquant / MiniQMT 适配层 | 第一版数据源和交易接口 |
| 测试工具 | pytest + Playwright | 后端单测、前端端到端测试 |
| 启动方式 | start.bat | 本地一键启动 |

### 5.1.1 当前第一版轻量实现偏差说明

截至 2026-05-10，当前项目的真实依赖以 `frontend/package.json` 为准：

```text
React + TypeScript + Vite
Ant Design v5 + @ant-design/icons
自研 DataTable / SectionCard / MetricCard / StatusChip 等轻量组件
Lightweight Charts
Monaco Editor
```

当前未安装：

```text
@ant-design/pro-components
echarts
```

这是已批准的轻量 MVP 偏差，原因是当前阶段要求“不引入新 UI 框架/不加重依赖/不大范围重构主链路”。因此，后续开发不得仅因为本总览文档提到 `ProTable / ProComponents / ECharts` 就直接新增依赖。

执行口径：

1. 新增表格优先复用自研 `DataTable`，不要在页面中散落原始 Ant Design Table。
2. 新增页面分区优先复用 `SectionCard`、`MetricCard`、`PageHeaderBar`、`EmptyGuide`、`LogDrawer`。
3. 图表在当前主链路未受影响时，优先保持现有实现；需要引入 ECharts 时必须另起明确阶段任务。
4. 只有当后续阶段明确要求 ProComponents 或 ECharts，且验证不会破坏性能、构建和主链路，才允许评估引入。

### 5.2 UI/UX 呈现方案

推荐前端组合：

```text
React + TypeScript + Vite
Ant Design + ProComponents
ECharts + Lightweight Charts
Monaco Editor
```

说明：以上是长期推荐组合，不等于当前第一版必须全部安装。当前第一版继续执行 `5.1.1 当前第一版轻量实现偏差说明`。

呈现目标：

1. 像成熟金融中后台，而不是脚本工具。
2. 页面清爽，信息密度高，但不拥挤。
3. 表格、图表、任务、日志都要专业可读。
4. 策略代码编辑体验接近 VS Code。
5. 错误提示对非程序员友好。

### 5.3 本地运行边界

第一版只要求 Windows 本地运行。

```text
start.bat
  ↓
启动后端 FastAPI
  ↓
启动前端 Vite 或本地静态服务
  ↓
自动打开浏览器 http://localhost:3000
```

开发阶段可以前后端分开启动；交付阶段提供一个一键启动脚本。

## 23. 需求冻结与变更规则

为了避免后续越改越乱，本项目必须先冻结第一版范围。

### 23.1 第一版冻结范围

第一版只允许开发以下主链路：

```text
系统启动
系统设置
环境检测
数据同步
数据展示
Python策略编辑
Python策略运行
策略信号展示
单策略回测
手动下单
信号下单
委托成交同步
日志备份
```

不属于以上链路的功能，默认不开发。

### 23.2 需求变更原则

新增需求必须满足至少一条：

1. 会影响主链路跑通。
2. 会影响交易安全。
3. 会影响数据正确性。
4. 会影响策略运行或回测可信度。
5. 会影响页面可操作性。
6. 会影响性能和稳定性。

如果只是“看起来更高级”，第一版不做。

### 23.3 模块优先级

| 优先级 | 模块/能力 | 原因 |
|---|---|---|
| P0 | 系统启动、环境检测、数据同步、策略运行、交易执行 | 没有这些主链路跑不通 |
| P1 | 回测、数据质量、日志、备份、运行监控 | 影响可信度和排错 |
| P2 | 图表优化、导出、历史版本对比 | 提升体验 |
| P3 | 多数据源、多账户、多策略组合、参数寻优 | 未来商业化再做 |

## 34. 最终定稿

本项目最终形态：

```text
产品名称：本地量化控制台
核心模块：总览看板、数据中心、策略开发、回测研究、交易执行、系统管理
策略方式：只使用 Python 新建策略
数据方式：QMT 同步到 SQLite，再供前端、策略、回测、交易使用
交易方式：手动下单和信号下单，统一经过交易执行模块
运行方式：本地一键启动
开发原则：先跑通主链路，再逐步增强
```

最关键的一句话：

> **后台不负责替用户生成策略逻辑；后台负责把 AI 写好的 Python 策略安全、稳定、可视化地接入数据、回测和交易流程。**
