# 交易执行

> 本文件是 `交易执行` 模块的独立规格。Codex 开发本模块时，应同时读取 `AGENTS.md`、`docs/00_项目总览.md`、通用架构规范和对应阶段计划。

## 13. 模块五：交易执行

### 13.1 模块定位

交易执行负责手动交易、信号下单、撤单、委托成交同步。

当前真实 QMT 运行口径是“只读验收优先”：页面保留手动下单、信号下单、撤单、订单生命周期和确认弹窗的完整流程，但在真实小额交易验收完成前，真实 QMT 下单与撤单能力必须由后端阻断。测试隔离交易只允许自动化测试和离线排障使用，不得混入默认业务视图。

第一版不做复杂风控，但必须保证订单生命周期清楚、状态可追踪、防重复点击。

### 13.2 页面结构

```text
交易执行
├── 交易面板
├── 信号下单
├── 当前持仓
├── 委托记录
├── 成交记录
└── 执行日志
```

### 13.3 交易面板

#### 前端功能

表单字段：

| 字段 | 说明 |
|---|---|
| 股票代码 | 输入或搜索 |
| 股票名称 | 自动补齐 |
| 买卖方向 | 买入 / 卖出 |
| 委托价格 | 限价或当前价 |
| 委托数量 | 股数，按 100 股检查 |
| 委托金额 | 自动计算 |
| 可用资金 | 自动展示 |
| 可卖数量 | 卖出时展示 |

按钮：

```text
买入
卖出
重置
```

交互：

1. 下单前弹窗确认。
2. 下单按钮防重复点击。
3. 下单后显示本地订单号。
4. 自动刷新委托记录。

#### 后端功能

服务：

```text
order_service.py
execution_service.py
qmt_trade_adapter.py
order_status_mapper.py
```

职责：

1. 校验参数。
2. 创建本地订单。
3. 通过交易适配器提交委托；真实小额验收前使用禁用型真实交易适配器阻断提交。
4. QMT 返回后保存 QMT 订单号；被阻断或失败时必须保留本地订单和中文错误。
5. 更新订单状态。
6. 写操作日志。

#### API

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/trading/orders/manual` | 手动下单 |
| POST | `/api/trading/orders/{order_id}/cancel` | 撤单 |

### 13.4 信号下单

#### 前端功能

展示未处理策略信号。

字段：

```text
信号时间、策略、股票、方向、参考价、建议金额、原因、状态、操作
```

操作：

```text
下单
忽略
查看K线
```

下单确认弹窗必须显示：

```text
策略名称
股票代码
股票名称
买卖方向
价格
数量
金额
信号原因
```

#### 后端功能

职责：

1. 读取信号。
2. 检查信号状态。
3. 创建信号订单。
4. 调用统一下单服务；真实 QMT 只读验收阶段不得绕过人工确认或后端阻断。
5. 更新信号状态为已下单。
6. 防止同一个信号重复下单。

#### API

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/trading/orders/from-signal/{signal_id}` | 信号下单 |
| POST | `/api/trading/signals/{signal_id}/ignore` | 忽略信号 |

### 13.8 订单生命周期

订单状态必须统一：

```text
待提交
已提交
已报
部分成交
全部成交
已撤
废单
失败
待同步
```

订单必须记录：

| 字段 | 说明 |
|---|---|
| local_order_id | 本地订单 ID |
| qmt_order_id | QMT 订单 ID |
| source | manual / signal / auto |
| signal_id | 信号 ID，可为空 |
| strategy_id | 策略 ID，可为空 |
| status | 内部订单状态 |
| qmt_status | QMT 原始状态 |
| idempotency_key | 幂等键，防重复下单 |

### 13.9 最小交易保护

这不是复杂风控，只是防误操作。

必须保留：

1. 顶部显示真实 QMT 只读 / 测试隔离状态。
2. 下单前确认弹窗。
3. 最大单笔金额设置。
4. 防重复点击。
5. 同一信号防重复下单。
6. 撤单前确认。
7. 交易操作日志。
8. 停止策略按钮。

### 13.10 交易性能与一致性

1. 下单按钮必须立即锁定。
2. 后端必须生成幂等键。
3. QMT 返回慢时不能重复提交。
4. 下单后必须先记录本地订单。
5. 委托和成交必须可同步修正状态。
6. 交易相关同步优先级高于历史行情同步。
7. QMT 异常时必须保留本地错误记录。

### 13.11 交易验收标准

当前只读验收标准：

1. 真实 QMT 只读状态下，手动下单、信号下单、撤单按钮不得提交真实委托。
2. 页面必须清楚提示“真实 QMT 只读验收中，真实下单需单独小额验收后再开启”。
3. 委托、成交、持仓和账户数据来自数据中心只读同步落库结果。
4. 能防止同一信号重复生成有效订单。
5. 订单状态完整可追踪。
6. 出错时能复制错误给 AI。

真实小额交易单独验收标准：

1. 用户人工确认后才能做小额手动买入/卖出。
2. 用户人工确认后才能从策略信号下单。
3. 用户人工确认后才能撤单。
4. 必须记录 `local_order_id`、`qmt_order_id`、订单状态流转和操作日志。
5. 必须同步委托状态和成交记录，并核对信号状态一致性。

## 28. 交易执行深度规范

### 28.1 下单流程

```text
前端点击下单
  ↓
前端弹窗确认
  ↓
前端按钮锁定
  ↓
后端生成 idempotency_key
  ↓
创建本地订单：待提交
  ↓
交易适配器提交委托
（真实小额验收前后端阻断）
  ↓
保存 qmt_order_id
  ↓
更新状态：已提交/失败
  ↓
前端提示结果
  ↓
后台同步委托成交
```

### 28.2 信号下单流程

```text
读取未处理信号
  ↓
用户点击下单
  ↓
检查信号是否已处理
  ↓
生成本地订单
  ↓
关联 signal_id 和 strategy_id
  ↓
交易适配器提交委托
（真实小额验收前后端阻断）
  ↓
更新信号状态为已下单
```

同一信号只能生成一个有效订单。

### 28.3 撤单流程

```text
用户点击撤单
  ↓
确认弹窗
  ↓
检查订单状态是否可撤
  ↓
交易适配器撤单
（真实小额验收前后端阻断）
  ↓
更新本地状态
  ↓
同步最新委托状态
```

### 28.4 订单状态映射

QMT 原始状态必须映射到内部状态。

内部状态固定：

```text
待提交
已提交
已报
部分成交
全部成交
已撤
废单
失败
待同步
```

状态映射必须集中在：

```text
order_status_mapper.py
```

禁止在前端或多个 service 里重复写映射逻辑。

### 28.5 幂等规则

幂等键组成建议：

```text
account_id + symbol + side + price + quantity + source + signal_id + 时间窗口
```

第一版至少做到：

1. 前端按钮锁定。
2. 后端检查 idempotency_key。
3. 同一 signal_id 不允许重复生成有效订单。
