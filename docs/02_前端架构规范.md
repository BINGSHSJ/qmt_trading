# 02_前端架构规范

> 本文件约束 React 前端工程结构、页面组织、请求封装、状态管理、组件复用和测试标识。

## 前端核心原则

1. 所有功能必须页面可操作。
2. 所有接口调用必须通过 `src/services/` 封装。
3. 不允许组件直接拼接后端 URL。
4. 不允许页面一次性加载全量数据。
5. 所有表格必须支持分页、筛选、排序和刷新。
6. 所有长任务必须通过 `task_id` 轮询进度。
7. 所有错误必须中文化，并提供技术详情弹窗。
8. Monaco、图表、大表格等重组件必须懒加载。
9. 页面必须包含加载态、空状态、错误态、成功反馈。
10. 主菜单和二级 Tab 必须四个汉字。

## 23. 当前第一版前端依赖边界

截至 2026-05-10，当前项目未安装 `@ant-design/pro-components` 和 `echarts`。这不是遗漏施工项，而是为了满足第一版轻量架构边界的批准偏差。

当前真实前端依赖以 `frontend/package.json` 为准：

| 类型 | 当前实现 |
|---|---|
| 前端框架 | React + TypeScript + Vite |
| UI 组件 | Ant Design v5 + `@ant-design/icons` |
| 高级表格 | 自研 `DataTable`，内部基于 Ant Design Table |
| 页面分区 | 自研 `SectionCard`、`MetricCard`、`PageHeaderBar` 等 |
| 图表 | 当前已跑通的轻量图表实现 + `lightweight-charts` |
| 代码编辑器 | Monaco Editor |

执行要求：

1. 后续新增表格必须优先使用 `DataTable`，不要在页面中直接新增散落的 Ant Design Table。
2. 不得为了“补齐 ProTable 文档”直接安装 `@ant-design/pro-components`。
3. 不得为了“补齐 ECharts 文档”直接安装 `echarts`。
4. 如未来阶段确实需要 ProComponents 或 ECharts，必须单独提出原因、影响范围、构建体积、性能验证和回滚方案。
5. 当前自研轻量组件属于第一版标准实现，后续应继续复用，而不是绕开重写。

## 24. 前端实现深度规范

### 24.1 前端页面交付标准

每个页面交付时必须同时具备：

| 能力 | 要求 |
|---|---|
| 页面标题 | 明确当前模块名称 |
| 操作区 | 高频按钮靠上显示 |
| 数据表格 | 分页、筛选、排序、刷新 |
| 加载状态 | 请求时显示 loading |
| 空状态 | 告知下一步操作 |
| 错误状态 | 中文说明 + 技术详情 |
| 最近更新时间 | 关键数据必须显示 |
| 响应反馈 | 操作成功/失败必须提示 |
| 任务入口 | 长任务可进入任务详情 |

### 24.2 前端服务层规范

所有 API 请求必须通过统一请求封装。

```text
services/request.ts
```

职责：

1. 统一 baseURL。
2. 统一超时设置。
3. 统一错误转换。
4. 统一 trace_id 记录。
5. 统一 loading 可选控制。
6. 统一下载处理。
7. 统一取消请求。

禁止页面组件直接写 `fetch` 或 `axios` 原始调用。

### 24.3 前端状态管理规范

第一版状态管理保持轻量。

推荐状态分类：

| 状态 | 存放位置 |
|---|---|
| 当前菜单/Tab | URL 或页面内部状态 |
| QMT 连接状态 | 全局 store |
| 当前任务状态 | 全局 store + 轮询 |
| 用户配置摘要 | 全局 store |
| 表格筛选条件 | 页面内部状态，必要时同步 URL |
| 代码编辑内容 | 页面内部状态 |

第一版不要引入复杂状态机，除非交易订单状态需要清晰表达。

### 24.4 前端组件规范

必须沉淀以下通用组件：

| 组件 | 用途 |
|---|---|
| StatusBar | 顶部状态栏 |
| PageHeader | 页面标题和操作区 |
| DataTable | 统一表格封装 |
| TaskProgress | 任务进度条 |
| ErrorDetailModal | 错误详情弹窗 |
| EmptyGuide | 空状态引导 |
| KLineChart | K 线图 |
| BacktestChart | 回测图表 |
| CodeEditor | Monaco 策略编辑器 |
| OrderConfirmModal | 下单确认弹窗 |
| LogViewer | 日志查看器 |
| ConfigForm | 设置表单 |

组件约束：

1. 组件必须有清晰 props 类型。
2. 组件不得直接调用业务 API，除非是页面级容器组件。
3. 通用组件不得写死模块数据。
4. 错误弹窗必须可复制技术详情。
5. 表格组件必须默认分页。

### 24.5 路由规范

路由固定：

```text
/dashboard
/data-center
/strategy-dev
/backtest
/trading
/system
```

Tab 状态建议使用 query 参数：

```text
/data-center?tab=数据同步
/strategy-dev?tab=代码编辑&strategy_id=1
```

好处：刷新页面后不丢状态，也方便从一个模块跳转到另一个模块。

### 24.6 表格筛选规范

所有表格筛选字段必须有默认折叠/展开策略。

默认展示：

```text
关键词
日期范围
状态
刷新按钮
```

高级筛选可以折叠：

```text
策略名称
股票代码
订单来源
任务类型
日志等级
```

### 24.7 前端任务轮询规范

长任务轮询统一封装 hook：

```text
useTaskPolling(taskId)
```

要求：

1. running 状态每 1-2 秒轮询。
2. success/failed/cancelled 后停止轮询。
3. 页面卸载时停止轮询。
4. 轮询失败超过次数后提示。
5. 支持手动刷新任务状态。

### 24.8 UI 文案规范

文案必须面向非程序员。

错误文案示例：

| 不允许 | 应改为 |
|---|---|
| DataFrame empty | 分钟K数据为空，请先同步分钟K数据 |
| KeyError symbol | 策略返回的信号缺少股票代码 symbol 字段 |
| xtquant import failed | xtquant 未安装或当前 Python 环境无法导入 |
| database locked | 数据库正在被其他任务占用，请稍后重试 |

### 24.9 前端测试标识

关键按钮和表格必须添加稳定测试标识，方便 Playwright 测试。

示例：

```text
data-testid="btn-sync-account"
data-testid="btn-run-strategy"
data-testid="table-strategy-signals"
data-testid="modal-order-confirm"
```
