# 01_UIUX设计规范

> 本文件约束前端呈现效果、交互方式、页面状态和金融后台视觉规范。Codex 开发任何页面时都必须遵守。

## 6. UI/UX 设计规范

### 6.1 设计目标

| 目标 | 要求 |
|---|---|
| 专业 | 界面像金融后台，而不是临时工具 |
| 简洁 | 不堆砌无用模块 |
| 高效 | 高频操作 1-2 次点击完成 |
| 清楚 | 数据状态、任务状态、交易状态一眼能看懂 |
| 可排错 | 出错后知道原因和下一步 |
| 可扩展 | 未来增加模块不破坏布局 |

### 6.2 页面布局

当前权威视觉方向采用“暗色金融桌面工作台”布局。参考 IBKR Desktop / TWS 类成熟交易终端的信息组织方式，但不复制品牌、不改变业务语义、不新增业务模块。

```text
AppShell
├── 总览看板
├── 数据中心
├── 策略开发
├── 回测研究
├── 交易执行
└── 系统管理

TopStatusBar
├── QMT 状态
├── 账户资产
├── 任务状态
├── 当前时间
└── 设置入口

PageCommandBar
├── 当前模块标题
├── 主操作
├── 刷新
└── 最近更新时间

Workbench
├── 主表格 / 主图表 / 主编辑区
├── 指标横条
├── 分组 Tab
└── 空态 / 错误态 / 加载态

Inspector / Drawer
├── 详情
├── 诊断
├── 日志
└── 技术信息

BottomStatusBar
├── 本地状态
├── SQLite 状态
├── 真实 QMT 只读
└── 自动实盘状态
```

### 6.3 顶部状态栏

顶部状态栏必须全局可见。

| 元素 | 说明 |
|---|---|
| QMT 状态 | 已连接 / 未连接 / 异常 |
| 交易状态 | 真实只读 / 测试隔离 / 未检测 |
| 账户资产 | 总资产或可用资金简写 |
| 任务状态 | 当前是否有运行任务 |
| 今日日期 | 交易日信息 |
| 快捷设置 | 进入系统管理 |

状态颜色：

| 状态 | 颜色建议 |
|---|---|
| 正常 | 绿色 |
| 运行中 | 蓝色 |
| 警告 | 黄色/橙色 |
| 异常 | 红色 |
| 未连接 | 灰色 |

### 6.4 主题规范

截至 2026-05-22，当前 UI/UX 视觉方向正式调整为暗色金融桌面工作台。旧的明亮主题口径不再作为当前施工标准。

| 项目 | 规范 |
|---|---|
| 背景色 | 深色终端背景，低饱和蓝黑 / 炭黑 |
| 主色 | 克制蓝色，用于主操作、选中态、可交互状态 |
| 强调色 | 盈利/上涨用红，亏损/下跌用绿，遵循 A 股习惯 |
| 状态色 | 正常绿、运行中蓝、警告橙、异常红、未知灰 |
| 分区 | 线框分区优先，少用大面积圆角卡片 |
| 圆角 | 工作台分区 0-4px；确认弹窗和少量强调面板可用 4-8px |
| 间距 | 4/6/8/12/16px 紧凑体系 |
| 字号 | 表格 12px，正文 12-13px，模块标题 14-16px |
| 数字 | 金额、价格、数量、时间使用等宽数字 |
| 表格密度 | 偏紧凑但可读，表格是主要信息载体 |

禁止：

1. 不做炫酷大屏。
2. 不使用大面积渐变装饰。
3. 不把页面做成营销站或卡片瀑布流。
4. 不为了视觉效果弱化交易确认、只读保护和错误诊断。
5. 不因为 IBKR 参考方向改变六大菜单、路由或业务流程。

### 6.4.1 暗色金融桌面 Token 建议

| Token | 建议 |
|---|---|
| `--lqc-terminal-bg` | 页面主背景 |
| `--lqc-terminal-panel` | 工作台面板背景 |
| `--lqc-terminal-panel-2` | 表头、命令条、次级面板 |
| `--lqc-terminal-line` | 默认分割线 |
| `--lqc-terminal-line-strong` | 强分割线和输入框边界 |
| `--lqc-terminal-text` | 主文本 |
| `--lqc-terminal-muted` | 次级说明 |
| `--lqc-terminal-blue` | 主操作、选中态 |
| `--lqc-profit-red` | A 股上涨 / 盈利 |
| `--lqc-loss-green` | A 股下跌 / 亏损 |

### 6.4.2 图表依赖口径

当前项目第一版按 `AGENTS.md` 的批准偏差执行：未安装 ECharts 时，不得为了补齐文档字样直接引入 ECharts。回测图表和 K 线图优先沿用当前已跑通实现；如后续确需引入 ECharts，必须单独评估依赖体积、性能、迁移成本和回滚方案。

### 6.4.3 设备工作区与显示密度

本项目是专业量化交易后台，不按普通 SaaS 大卡片后台设计。视觉密度必须通过真实响应式布局和 Token 控制实现，不允许要求用户把浏览器缩放到 80%。

目标工作区：

| 设备 | 目标有效工作区 | 要求 |
|---|---|---|
| MacBook Pro 14 寸 M1/M2 | `1512 x 982` | 不能拥挤；左侧导航、顶部状态、命令区、主表格必须可同时阅读 |
| 27 寸 Windows | `2560 x 1440` | 元素不得放大成普通后台；表格和工作台应增加信息承载 |
| 27 寸 Windows 4K 125% | 约 `3072 x 1728` 有效工作区 | 不用大面积空白填充；指标、命令卡和详情面板必须有最大宽度约束 |

密度规则：

1. 默认密度为 `compact`。
2. 保留 `comfortable / compact / dense` 三档密度 Token。
3. `html[data-density]` 控制 CSS 变量，Ant Design `ConfigProvider` 只负责组件基础 Token。
4. 14 寸 Mac 视口使用更紧凑的侧栏、顶部栏、内容间距和表格高度。
5. 27 寸 Windows 视口不得简单放大字号、按钮和卡片，应通过更多列、右侧检查器和表格承载信息。
6. 不使用 `body zoom`、`transform scale`、外部字体文件或浏览器缩放作为解决方案。

### 6.5 表格规范

所有表格必须具备：

1. 分页。
2. 搜索。
3. 筛选。
4. 排序。
5. 刷新。
6. 列宽合理。
7. 空状态。
8. 加载状态。
9. 错误状态。
10. 最近更新时间。

表格分页规则：

| 项目 | 规则 |
|---|---|
| 默认页数 | 20 或 50 |
| 最大页数 | 200 |
| 禁止 | 一次性加载上万条 |
| 导出 | 单独走导出任务，不走列表接口 |

### 6.6 表单规范

表单必须做到：

1. 字段有中文标签。
2. 字段有默认值。
3. 必填字段有标识。
4. 输入错误时有中文提示。
5. 金额、数量、日期、路径要有专用控件。
6. 提交后按钮进入 loading。
7. 成功后给出反馈并刷新数据。

### 6.7 弹窗规范

危险或重要操作必须弹窗确认：

| 操作 | 弹窗要求 |
|---|---|
| 下单 | 显示股票、方向、价格、数量、金额 |
| 撤单 | 显示委托号、股票、数量 |
| 删除策略 | 显示策略名称，确认不可恢复 |
| 恢复备份 | 显示恢复内容和风险 |
| 停止任务 | 显示当前任务和影响 |

第一版不做复杂风控，但下单确认必须保留。

### 6.8 图表规范

| 图表 | 工具 | 要求 |
|---|---|---|
| 收益曲线 | ECharts | 支持缩放、悬浮提示 |
| 回撤曲线 | ECharts | 与收益曲线风格一致 |
| 每日盈亏 | ECharts | 可选第一版后置 |
| K 线图 | Lightweight Charts | 支持日 K、分钟 K |
| 任务耗时 | ECharts | 系统管理运行监控使用 |

图表约束：

1. 不一次渲染过多点，超过阈值要降采样或分页加载。
2. 图表容器必须有明确高度。
3. 图表加载失败要显示中文错误。
4. K 线图只用于查看，不做复杂行情终端。

### 6.9 代码编辑规范

策略代码编辑器使用 Monaco Editor。

必须支持：

1. Python 语法高亮。
2. 保存。
3. 另存为。
4. 检查策略接口。
5. 试运行。
6. 显示错误行号。
7. 复制错误给 AI。
8. 查看版本记录。
9. 恢复历史版本。

代码编辑器约束：

1. Monaco 必须懒加载，不得影响首页加载速度。
2. 保存前必须提醒是否覆盖当前版本。
3. 每次保存必须生成策略版本快照。
4. 不允许 Codex 自动覆盖用户策略目录。

### 6.10 错误提示规范

错误提示分两层。

第一层：用户可读中文提示。

```text
策略运行失败：分钟 K 数据为空，请先到数据中心同步分钟 K 数据。
```

第二层：技术详情。

```text
KeyError: minute_kline
Traceback: ...
```

所有错误弹窗必须提供：

```text
复制中文说明
复制技术详情
复制全部给 AI
```

### 6.11 空状态规范

空状态不能只显示“暂无数据”，必须告诉用户下一步。

示例：

```text
暂无持仓数据，请先点击“同步持仓”。
```

```text
暂无策略文件，请点击“新建策略”或“导入策略”。
```

```text
暂无分钟 K 数据，请到数据同步中同步分钟 K。
```

### 6.12 加载状态规范

所有长操作必须可见：

| 操作 | 加载表现 |
|---|---|
| 同步数据 | 任务进度条 + 日志 |
| 运行策略 | 状态标签 + 停止按钮 |
| 启动回测 | 进度条 + 当前阶段 |
| 下单 | 按钮锁定 + 结果提示 |
| 撤单 | 按钮锁定 + 状态更新 |
| 备份恢复 | 进度条 + 完成提示 |

## 30. UI 视觉细化规范

### 30.1 页面信息密度

金融后台信息密度可以偏高，但要分层：

1. 顶部状态栏显示全局状态。
2. 页面命令栏显示当前模块主操作。
3. 指标横条显示摘要，不使用大量大卡片。
4. 表格显示明细，保持列宽、分页和筛选稳定。
5. 右侧检查器或抽屉显示详情、日志和诊断。
6. 弹窗只处理确认、危险操作和强提示。
7. 底部状态栏显示本地运行状态，不承载主业务操作。

### 30.2 页面默认状态

首次打开各页面时：

| 页面 | 默认状态 |
|---|---|
| 总览看板 | 账户 / 信号 / 任务 / 交易摘要工作台 |
| 数据中心 | 数据新鲜度、同步落库、覆盖质量和账户行情入口 |
| 策略开发 | 策略文件列表、代码编辑、运行调试和版本记录 |
| 回测研究 | 回测任务、绩效结果、交易明细和日志证据 |
| 交易执行 | 信号下单、人工确认、委托、成交和执行日志 |
| 系统管理 | 基础设置、环境检测、运行监控、日志和备份 |

### 30.3 关键颜色语义

| 语义 | 颜色 |
|---|---|
| 成功/正常 | 绿色 |
| 失败/危险 | 红色 |
| 警告/待处理 | 橙色 |
| 运行中 | 蓝色 |
| 停用/未知 | 灰色 |

A 股盈亏颜色：

| 状态 | 颜色 |
|---|---|
| 盈利/上涨 | 红色 |
| 亏损/下跌 | 绿色 |

### 30.4 操作按钮层级

| 层级 | 用途 |
|---|---|
| Primary | 主要操作，如同步、运行、下单 |
| Default | 普通操作，如查看、刷新 |
| Danger | 删除、恢复备份、撤单、停止任务 |
| Link | 跳转详情 |

### 30.5 详情展示方式

| 内容 | 展示方式 |
|---|---|
| 任务日志 | 右侧抽屉或检查器 |
| 订单详情 | 右侧抽屉；危险操作仍用确认弹窗 |
| 错误详情 | 弹窗 |
| 策略版本对比 | 全屏弹窗或独立页面 |
| K 线图 | 工作台图表区或抽屉 |

### 30.6 六大模块工作台模型

| 模块 | 工作台模型 | 视觉重点 |
|---|---|---|
| 总览看板 | 三栏摘要工作台 | 账户、信号、任务、交易摘要和主链路入口 |
| 数据中心 | 数据运营工作台 | 数据源、覆盖率、同步任务、质量、字典分层 |
| 策略开发 | 轻量 IDE 工作台 | 文件列表、代码编辑器、运行结果、版本和错误 |
| 回测研究 | 回测实验室工作台 | 任务、曲线、指标、交易明细、可信证据 |
| 交易执行 | Trading Blotter 工作台 | 信号、委托、成交、持仓、确认弹窗和状态流 |
| 系统管理 | 本地运维工作台 | 配置、环境检测、日志、备份、监控和操作记录 |

### 30.7 页面骨架验收

每个页面必须满足：

1. 左侧六大菜单名称保持不变。
2. 顶部全局状态栏可见。
3. 当前页面主操作集中在 PageCommandBar。
4. 主数据区优先使用表格、图表或编辑器，不堆无意义卡片。
5. 详情、日志、技术信息进入右侧检查器、抽屉或标准弹窗。
6. 空态、错误态、加载态都有中文说明和下一步建议。
7. 1280x720、1440x900、1920x1080 三档视口无页面级横向溢出。
8. 100% 浏览器缩放下不需要用户手动缩放到 80%。
9. 页面不得出现默认 Ant Design 白底块、浅色输入框泄漏或底部异常横条。
10. 交易确认、备份恢复、错误详情等弹窗必须使用统一尺寸和按钮区。
