# 03_后端架构规范

> 本文件约束 FastAPI 后端分层、统一响应、异常处理、QMT Adapter、任务执行器、策略隔离和日志规范。

## 7. 通用后端规范

### 7.1 API 响应格式

所有 API 统一响应：

```json
{
  "success": true,
  "message": "操作成功",
  "data": {},
  "error": null,
  "trace_id": "20260508-xxxx"
}
```

错误响应：

```json
{
  "success": false,
  "message": "分钟K数据为空，请先同步分钟K数据。",
  "data": null,
  "error": {
    "code": "MARKET_DATA_EMPTY",
    "detail": "minute_kline has no rows for symbol=600000.SH",
    "suggestion": "请到数据中心同步分钟K数据"
  },
  "trace_id": "20260508-xxxx"
}
```

### 7.2 分页格式

列表接口统一：

请求参数：

```text
page=1
page_size=50
sort_field=created_at
sort_order=desc
keyword=
start_date=
end_date=
```

返回格式：

```json
{
  "items": [],
  "page": 1,
  "page_size": 50,
  "total": 0,
  "has_more": false
}
```

### 7.3 任务状态格式

```json
{
  "task_id": "task_20260508_xxxx",
  "task_type": "sync_minute_kline",
  "status": "running",
  "progress": 45,
  "message": "正在同步分钟K：600000.SH",
  "started_at": "2026-05-08 10:00:00",
  "finished_at": null,
  "error_message": null,
  "technical_detail": null
}
```

### 7.4 日志格式

日志必须包含：

| 字段 | 说明 |
|---|---|
| log_id | 日志 ID |
| module | 模块 |
| level | info/warning/error |
| message | 中文信息 |
| technical_detail | 技术详情 |
| related_id | 关联任务/策略/订单 |
| created_at | 时间 |

### 7.5 异常处理规范

后端必须提供统一异常类：

| 异常类型 | 示例 |
|---|---|
| ConfigError | QMT 路径未配置 |
| QmtConnectionError | QMT 未连接 |
| DataSyncError | 数据同步失败 |
| StrategyValidationError | 策略接口不正确 |
| StrategyRunError | 策略运行失败 |
| BacktestError | 回测失败 |
| TradingError | 下单失败 |
| DatabaseError | 数据库写入失败 |

API 层不得直接返回 Python 原始异常。

## 25. 后端实现深度规范

### 25.1 后端分层职责

| 层 | 职责 | 禁止 |
|---|---|---|
| API | 参数校验、调用服务、返回响应 | 写业务逻辑、直接 SQL、直接 QMT |
| Schema | 请求响应模型 | 写业务逻辑 |
| Service | 业务编排 | 直接拼复杂 SQL、直接返回 ORM 对象 |
| Repository | 数据库读写 | 调用 QMT、写业务判断 |
| Adapter | 外部接口适配 | 直接写前端响应 |
| Task | 长任务执行 | 操作 UI 状态 |
| Core | 配置、异常、日志、响应 | 写具体业务 |

### 25.2 Repository 规范

Repository 必须提供：

1. 分页查询。
2. 按 ID 查询。
3. 批量 upsert。
4. 事务处理。
5. 常用条件查询。

禁止 Service 层到处写 SQL 字符串。

### 25.3 事务规范

以下操作必须使用事务：

1. 批量同步 K 线。
2. 保存策略信号。
3. 创建本地订单。
4. 更新订单和信号状态。
5. 保存回测结果和交易明细。
6. 备份恢复前的状态更新。

交易下单特殊要求：

```text
创建本地订单 → 提交 QMT → 更新 QMT 订单号 → 更新状态
```

如果 QMT 提交失败，也必须保留本地失败记录。

### 25.4 错误码规范

错误码统一大写英文，前端展示中文 message。

| 错误码 | 中文说明 |
|---|---|
| QMT_NOT_CONNECTED | QMT 未连接 |
| QMT_PATH_INVALID | QMT 路径无效 |
| XTQUANT_IMPORT_FAILED | xtquant 无法导入 |
| DATA_EMPTY | 数据为空 |
| DATA_SYNC_FAILED | 数据同步失败 |
| STRATEGY_FILE_NOT_FOUND | 策略文件不存在 |
| STRATEGY_INTERFACE_INVALID | 策略接口不正确 |
| STRATEGY_RUN_FAILED | 策略运行失败 |
| BACKTEST_DATA_MISSING | 回测数据缺失 |
| BACKTEST_FAILED | 回测失败 |
| ORDER_DUPLICATED | 重复下单 |
| ORDER_SUBMIT_FAILED | 下单失败 |
| ORDER_CANCEL_FAILED | 撤单失败 |
| DATABASE_LOCKED | 数据库被占用 |
| TASK_NOT_FOUND | 任务不存在 |

### 25.5 配置管理规范

配置来源优先级：

```text
默认配置 → app_config 表 → 环境变量或启动参数
```

第一版主要通过后台页面写入 `app_config`。

配置不要散落在多个 Python 文件里。

### 25.6 日志规范

日志分两类：

1. 文件日志：开发和技术排查。
2. 数据库日志：前端页面展示。

文件日志目录：

```text
logs/app.log
logs/error.log
logs/strategy.log
logs/backtest.log
logs/trading.log
```

数据库日志用于页面查询，不应记录无限长 traceback。长 traceback 可以写文件，数据库保存摘要和文件路径。

### 25.7 QMT Adapter 规范

QMT 调用统一封装，不允许散落。

接口建议：

```python
class QmtAdapter:
    def check_environment(self): ...
    def connect(self): ...
    def disconnect(self): ...
    def get_account(self): ...
    def get_positions(self): ...
    def get_orders(self): ...
    def get_trades(self): ...
    def get_stock_basic(self): ...
    def get_daily_kline(self, symbols, start_date, end_date): ...
    def get_minute_kline(self, symbols, start_time, end_time, period="1m"): ...
    def place_order(self, order_request): ...
    def cancel_order(self, qmt_order_id): ...
```

同时必须提供测试隔离替身：

```python
class TestIsolationQmtDataAdapter:
    ...
```

用于自动化测试、离线回归和开发排障；业务运行模式不得默认使用 Mock，也不得在真实 QMT 不可用时自动回退测试隔离数据。

### 25.8 任务执行器规范

第一版可以使用线程池或进程池，不引入 Celery。

任务执行器必须支持：

1. 创建任务。
2. 更新进度。
3. 追加日志。
4. 标记成功。
5. 标记失败。
6. 标记取消。
7. 查询任务状态。

策略运行和回测建议使用独立进程或可中断执行方式，避免阻塞主服务。

## 26. 策略运行安全与隔离规范

### 26.1 基本原则

用户策略是本地 Python 代码。第一版不承诺完全沙箱化，但必须做最小隔离，避免策略误操作交易和系统资源。

### 26.2 必须隔离

1. 策略拿不到 QMT 原始连接。
2. 策略拿不到交易下单服务。
3. 策略拿不到数据库连接对象。
4. 策略通过 `StrategyContext` 读数据。
5. 策略只能返回信号。
6. 策略运行有超时。
7. 策略运行日志被捕获。
8. 策略运行失败不影响主服务。

### 26.3 建议限制

1. 策略在子进程中运行。
2. 策略工作目录限制在项目目录内。
3. 不把系统配置和账户敏感信息传给策略。
4. 对策略输出做严格校验。
5. 策略信号金额超过系统最大单笔金额时，交易执行模块再次确认或拦截。

### 26.4 明确风险提示

由于 Python 本身可执行任意本地代码，用户从 AI 或外部复制策略代码前，应在代码编辑器中检查。系统可以提供“策略代码安全提示”，但第一版不做完整恶意代码检测。

## 分层硬约束

| 层级 | 职责 | 禁止 |
|---|---|---|
| API | 参数接收、权限预留、调用 Service | 直接写业务逻辑、直接操作数据库 |
| Schema | 请求与响应结构 | 混入数据库逻辑 |
| Service | 业务流程编排 | 直接散落 QMT 调用 |
| Repository | 数据库读写 | 写复杂业务决策 |
| Adapter | 外部系统适配，如 QMT | 被前端或策略直接调用 |
| Task | 长任务执行、进度、日志 | 阻塞 API 请求 |
| Core | 配置、日志、异常、工具 | 放业务模块逻辑 |
