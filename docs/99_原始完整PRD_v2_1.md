# 本地量化控制台｜Codex 开发需求文档 v2.0

> 文档用途：交给 Codex / GPT-5.5 / 其他大模型开发工具，作为本地量化控制台的唯一主需求文档。  
> 当前版本：v2.1 深度规格增强版  
> 日期：2026-05-08  
> 项目定位：个人本地使用，未来可商业化扩展。  
> 核心目标：**数据稳定同步、Python 策略可开发、回测结果可信、交易执行可控、后台全程可视化。**

---

## 0. 版本总说明

本版本是在 v1.1 的基础上继续收敛需求，重点解决以下问题：

1. 防止后续越改越乱。
2. 明确每个模块的功能边界。
3. 明确前端、后端、数据库、任务、性能、UI/UX 的实现约束。
4. 明确策略只使用 Python 新建策略，不做筛选器、拖拽器、条件组合器。
5. 明确第一版是轻量 MVP，但架构命名和模块边界按未来可商业化产品设计。
6. 明确所有核心操作都必须在可视化后台完成，用户不需要打开命令行、数据库或配置文件。

本文件是项目第一版开发的**主规格书**。Codex 开发时必须以本文件为准，不得自行扩展重功能。

## 1. 给 Codex 的总指令

### 1.1 必须遵守

1. 所有核心功能必须通过后台页面操作。
2. 用户不需要手动运行 Python 脚本。
3. 用户不需要打开 SQLite 数据库。
4. 用户不需要手动修改 JSON 配置文件。
5. 用户不需要看英文报错。
6. 所有错误必须有中文解释，同时保留技术详情，方便复制给 AI 排查。
7. 主菜单和二级 Tab 菜单统一使用 **四个汉字**。
8. 策略只支持 **Python 新建策略**。
9. 不做条件筛选建策略。
10. 不做拖拽策略。
11. 不做无代码策略生成器。
12. QMT 只是第一版数据源，架构上不要写死成只能支持 QMT。
13. 所有数据先落本地 SQLite，再给前端、策略、回测和交易使用。
14. Python 策略只生成信号，不直接调用 QMT 下单。
15. 下单统一经过交易执行模块。
16. Codex 不得随意覆盖 `strategies/user/` 目录中的用户策略文件。
17. 所有长任务必须返回 `task_id`，前端通过任务状态查询进度。
18. 所有列表接口必须分页、筛选、排序，禁止一次性返回全量数据。
19. 所有页面必须有加载态、空状态、错误态、刷新按钮、最近更新时间。
20. 第一版不追求功能多，追求流程完整、稳定、清晰、可维护。

### 1.2 明确不做

| 不做内容 | 原因 |
|---|---|
| 多用户系统 | 当前本地个人使用 |
| 登录权限 | 第一版避免复杂化 |
| 角色管理 | 第一版无必要 |
| 多租户 | 商业化后再做 |
| 复杂风控中心 | 用户明确不需要重风控 |
| 审批流 | 个人使用无意义 |
| 条件筛选建策略 | 已取消，统一 Python 策略 |
| 拖拽策略 | 开发成本高，且不适合真实策略 |
| AI 工作台 | 暂不需要 |
| 消息中心 | 暂不需要 |
| 商业化报表 | 后续再做 |
| 参数网格寻优 | 第一版不做 |
| 多策略组合 | 第一版不做 |
| Redis | 第一版不引入 |
| Celery | 第一版不引入 |
| Docker | 第一版不作为默认启动方式 |
| 微服务 | 第一版必须单体架构 |
| Kubernetes | 完全不需要 |
| 云端部署 | 第一版本地运行 |

### 1.3 第一版必须完整跑通的主链路

```text
启动系统
  ↓
系统管理完成 QMT 环境检测
  ↓
数据中心同步 QMT 账户、持仓、委托、成交、行情
  ↓
策略开发中新建或导入 Python 策略
  ↓
策略运行产生信号
  ↓
回测研究使用同一策略进行回测
  ↓
交易执行支持手动下单和信号下单
  ↓
同步委托成交结果
  ↓
总览看板展示状态、资产、信号、交易、任务
```

## 2. 产品定位

### 2.1 产品名称

中文名称：**本地量化控制台**  
英文代号：`Local Quant Console`

### 2.2 产品定位

本项目是一个本地运行的 A 股量化操作系统，当前只给个人使用，但模块命名、代码结构和数据结构要能支持未来商业化扩展。

它不是行情终端，也不是大型多用户量化平台。第一版只围绕四件事：

```text
数据同步 → 策略开发 → 回测研究 → 交易执行
```

### 2.3 用户画像

| 项目 | 描述 |
|---|---|
| 当前用户 | 个人投资者，主要使用 QMT/MiniQMT |
| 代码能力 | 不懂代码，但可以让 AI 写 Python 策略 |
| 操作习惯 | 希望全部在可视化后台完成 |
| 策略方式 | Python 策略文件，由 AI 辅助编写 |
| 数据来源 | 第一版以 QMT/MiniQMT 为主 |
| 部署方式 | 本地电脑运行 |
| 未来方向 | 未来可能商业化，扩展多数据源、多账户、多策略 |

### 2.4 核心设计原则

| 原则 | 说明 |
|---|---|
| 化繁为简 | 只保留当前确实需要的功能 |
| 名字正规 | 模块命名按产品化系统设计 |
| 功能轻量 | 不做企业级重系统 |
| 页面优先 | 不懂代码也能操作 |
| 策略代码化 | 策略统一使用 Python 文件 |
| 数据先落库 | 所有数据统一入库再使用 |
| 长任务可见 | 同步、策略、回测、备份都要有进度 |
| 交易可追踪 | 下单、撤单、成交必须有记录 |
| 错误可排查 | 中文说明 + 技术详情 + 复制给 AI |
| 性能可控 | 避免全量加载、阻塞页面、无索引查询 |
| 未来可扩展 | 当前轻量，但边界清晰 |

## 4. 总体架构设计

### 4.1 架构总览

第一版采用本地单体架构，不做微服务，不引入 Redis/Celery/Docker 作为默认依赖。

```text
React + TypeScript 前端
        ↓ HTTP API
FastAPI 后端
        ↓
Service 服务层
        ↓
Repository 数据访问层
        ↓
SQLite 本地数据库
        ↓
QMT / MiniQMT / xtquant 适配层
```

### 4.2 前端架构

前端采用页面模块化架构。

```text
frontend/src/
├── app/                 # 路由、布局、全局入口
├── pages/               # 6 个主菜单页面
├── components/          # 通用组件
├── services/            # API 请求封装
├── stores/              # 轻量全局状态
├── hooks/               # 通用 hooks
├── utils/               # 工具函数
├── types/               # TypeScript 类型
└── theme/               # UI 主题变量
```

前端必须遵守：

1. 页面通过路由懒加载。
2. Monaco Editor、ECharts、Lightweight Charts 必须懒加载。
3. 所有接口统一经过 `services/request.ts`。
4. 所有接口错误统一转换为中文错误提示。
5. 表格统一使用分页，不允许一次加载全量。
6. 长任务统一显示进度条、状态、日志入口。
7. 所有危险操作使用确认弹窗。
8. 页面刷新不能丢失当前 Tab、筛选条件和任务状态。
9. 任何页面都不能依赖浏览器控制台排查错误。

### 4.3 后端架构

后端采用 API → Service → Repository → Adapter 分层。

```text
backend/
├── api/                 # FastAPI 路由层，只负责参数校验和响应
├── schemas/             # Pydantic 请求/响应模型
├── services/            # 业务服务层
├── repositories/        # 数据库访问层
├── adapters/            # QMT 等外部适配层
├── tasks/               # 长任务执行器
├── core/                # 配置、日志、异常、依赖注入
├── models/              # ORM 或 SQLModel 模型
└── utils/               # 工具函数
```

后端必须遵守：

1. API 层不得写业务逻辑。
2. Service 层不得直接拼 SQL，统一走 Repository。
3. QMT 调用只能出现在 Adapter 层。
4. 策略代码不能拿到 QMT 原始对象。
5. 交易下单必须经过 Trading Service。
6. 所有异常必须转成统一错误结构。
7. 所有写操作必须记录 `operation_log`。
8. 所有长任务必须写入 `runtime_task`。
9. 所有任务日志必须写入对应日志表或统一日志文件。
10. 所有数据库写入必须使用事务。

### 4.4 数据架构

核心原则：**数据中心是数据入口，SQLite 是单一事实来源。**

```text
QMT 原始数据
  ↓
Adapter 标准化
  ↓
Repository 批量 upsert
  ↓
SQLite 标准表
  ↓
前端 / 策略 / 回测 / 交易读取
```

禁止：

1. 前端直接调用 QMT。
2. 策略直接调用 QMT。
3. 回测直接调用 QMT。
4. 交易状态绕过数据库只看接口返回。
5. 同一份业务数据多处重复维护。

### 4.5 任务架构

同步、策略运行、回测、环境检测、备份恢复都属于长任务。

统一任务流程：

```text
前端点击按钮
  ↓
POST 创建任务
  ↓
后端立即返回 task_id
  ↓
后台线程/任务执行器执行
  ↓
runtime_task 更新进度
  ↓
前端轮询任务状态
  ↓
任务完成后跳转结果或刷新页面
```

任务必须支持：

| 能力 | 要求 |
|---|---|
| 任务状态 | pending / running / success / failed / cancelled |
| 任务进度 | 0-100，可估算也可以阶段式 |
| 任务日志 | 可实时追加和查看 |
| 任务取消 | 策略运行、回测、长同步必须可取消 |
| 失败原因 | 中文原因 + 技术详情 |
| 重试能力 | 同步、环境检测、备份可重试 |

### 4.6 模块边界

| 模块 | 负责 | 不负责 |
|---|---|---|
| 总览看板 | 展示摘要、快捷入口 | 不承载复杂配置 |
| 数据中心 | 数据源、同步、质量、字典 | 不做下单 |
| 策略开发 | Python 策略文件、运行、信号 | 不直接下单 |
| 回测研究 | 使用策略和历史数据做验证 | 不调用 QMT 实盘交易 |
| 交易执行 | 手动/信号下单、撤单、状态同步 | 不写策略逻辑 |
| 系统管理 | 设置、检测、日志、备份 | 不做业务交易决策 |

## 5. 技术选型

### 5.1 总体选型

| 层级 | 技术 | 说明 |
|---|---|---|
| 前端框架 | React + TypeScript | 类型清晰，适合中后台 |
| 构建工具 | Vite | 本地开发快，配置轻 |
| UI 组件 | Ant Design | 中后台组件完整 |
| 高级表格 | ProTable / Ant Design Table | 数据表格多，适合后台 |
| 图表 | ECharts | 绩效、任务、统计图 |
| K 线图 | Lightweight Charts | 金融 K 线展示 |
| 代码编辑器 | Monaco Editor | Python 策略编辑 |
| 后端框架 | FastAPI | Python API 开发效率高 |
| 数据库 | SQLite | 本地个人版足够，低复杂度 |
| 数据处理 | pandas / numpy | 策略和回测计算 |
| QMT 对接 | xtquant / MiniQMT 适配层 | 第一版数据源和交易接口 |
| 测试工具 | pytest + Playwright | 后端单测、前端端到端测试 |
| 启动方式 | start.bat | 本地一键启动 |

### 5.2 UI/UX 呈现方案

推荐前端组合：

```text
React + TypeScript + Vite
Ant Design + ProComponents
ECharts + Lightweight Charts
Monaco Editor
```

呈现目标：

1. 像成熟金融中后台，而不是脚本工具。
2. 页面清爽，信息密度高，但不拥挤。
3. 表格、图表、任务、日志都要专业可读。
4. 策略代码编辑体验接近 VS Code。
5. 错误提示对非程序员友好。

### 5.3 本地运行边界

第一版只要求 Windows 本地运行。

```text
start.bat
  ↓
启动后端 FastAPI
  ↓
启动前端 Vite 或本地静态服务
  ↓
自动打开浏览器 http://localhost:3000
```

开发阶段可以前后端分开启动；交付阶段提供一个一键启动脚本。

### 6.6 表单规范

表单必须做到：

1. 字段有中文标签。
2. 字段有默认值。
3. 必填字段有标识。
4. 输入错误时有中文提示。
5. 金额、数量、日期、路径要有专用控件。
6. 提交后按钮进入 loading。
7. 成功后给出反馈并刷新数据。

### 6.7 弹窗规范

| 操作 | 弹窗要求 |
|---|---|

| 恢复备份 | 显示恢复内容和风险 |

### 6.8 图表规范

### 6.9 代码编辑规范

3. 另存为。

5. 试运行。
6. 显示错误行号。
7. 复制错误给 AI。
8. 查看版本记录。

## 8. 通用数据库规范

### 8.1 SQLite 规范

第一版使用 SQLite，但要从一开始按性能规范设计。

必须配置：

```sql
PRAGMA journal_mode = WAL;
PRAGMA synchronous = NORMAL;
PRAGMA busy_timeout = 5000;
PRAGMA foreign_keys = ON;
```

### 8.2 表设计通用字段

所有业务表建议包含：

| 字段 | 说明 |
|---|---|
| id | 主键 |
| created_at | 创建时间 |
| updated_at | 更新时间 |
| deleted_at | 可选，软删除 |
| remark | 可选备注 |

高频数据表如 K 线表可以不使用软删除，以性能为优先。

### 8.3 时间字段规范

| 字段 | 规范 |
|---|---|
| 日期 | `YYYY-MM-DD` |
| 时间 | `YYYY-MM-DD HH:mm:ss` |
| K 线时间 | 必须统一交易所时区，本地按北京时间处理 |
| 存储类型 | SQLite 中可用 TEXT，必须格式统一 |

### 8.4 股票代码规范

统一使用：

```text
600000.SH
000001.SZ
```

不要混用：

```text
600000
SH600000
600000.XSHG
```

如 QMT 返回格式不同，必须在 Adapter 层标准化。

### 8.5 批量写入规范

1. 同步数据必须批量 upsert。
2. 禁止逐条 commit。
3. 每批建议 500-5000 条，视数据类型调整。
4. 同步任务失败时要记录断点。
5. 使用 `sync_cursor` 记录同步位置。

### 8.6 索引规范

必须给以下字段建立索引：

| 表 | 索引字段 |
|---|---|
| daily_kline | symbol, trade_date |
| minute_kline | symbol, datetime |
| strategy_signal | strategy_id, signal_time, status |
| order_record | order_time, symbol, status |
| trade_record | trade_time, symbol |
| runtime_task | task_type, status, created_at |
| operation_log | module, created_at |
| sync_log | sync_type, status, started_at |

### 13.10 交易性能与一致性

1. 下单按钮必须立即锁定。
2. 后端必须生成幂等键。
3. QMT 返回慢时不能重复提交。
4. 下单后必须先记录本地订单。
5. 委托和成交必须可同步修正状态。
6. 交易相关同步优先级高于历史行情同步。
7. QMT 异常时必须保留本地错误记录。

### 13.11 交易验收标准

6. 能防止同一信号重复下单。
7. 订单状态完整可追踪。
8. 出错时能复制错误给 AI。

## 18. 性能优化与容量规划

### 18.1 性能红线

禁止以下实现：

1. 前端全量加载大表。
2. 回测接口同步等待直到结束。
3. 策略运行阻塞 API 主线程。
4. 分钟 K 默认全市场多年同步。
5. SQLite 逐条 commit。
6. 查询无索引。
7. 日志无限增长。
8. 图表一次渲染几十万点。
9. QMT 调用散落在多个模块。
10. 同一信号重复下单。

### 18.2 容量规划

| 数据 | 第一版策略 |
|---|---|
| 股票基础 | 全量同步 |
| 日 K | 可较长周期同步 |
| 分钟 K | 按股票池和时间区间同步 |
| 账户持仓 | 高频同步但数据小 |
| 委托成交 | 按日期增量同步 |
| 策略信号 | 分页展示，定期清理 |
| 回测明细 | 按任务保存，可删除 |
| 日志 | 保留天数可配置 |

### 18.3 前端性能

1. 路由懒加载。
2. 大组件懒加载。
3. 表格分页。
4. 搜索防抖 300ms。
5. 自动刷新间隔最低 5 秒。
6. 离开页面取消请求。
7. 图表数据降采样。
8. Monaco 只在代码编辑 Tab 加载。

### 18.4 后端性能

1. 读写分层。
2. 数据批量写入。
3. 查询加索引。
4. 长任务后台执行。
5. API 不返回过大 payload。
6. 导出单独任务化。
7. 任务日志分批写入。
8. QMT 慢调用有超时控制。

### 18.5 SQLite 性能

1. 启用 WAL。
2. 批量 upsert。
3. 使用唯一索引防重复。
4. 大表按 symbol/date 查询。
5. 定期清理日志和旧任务。
6. 提供数据库大小监控。
7. 必要时提供 VACUUM/维护按钮，但默认不自动频繁执行。

### 18.6 回测性能

1. 回测数据按需加载。
2. 单策略单任务。
3. 分批保存交易和曲线。
4. 支持取消。
5. 支持进度。
6. 分钟级回测必须限制范围。

### 18.7 交易性能

1. 交易接口优先级最高。
2. 下单请求不能被历史行情同步阻塞。
3. 防重复提交。
4. 委托状态同步必须及时。
5. 订单异常必须可追踪。

## 19. 测试与验收规范

### 19.1 后端测试

使用 pytest。

必须覆盖：

1. 配置读取保存。
2. 数据同步任务创建。
3. 批量写入和 upsert。
4. 策略接口检查。
5. 策略运行信号保存。
6. 回测任务创建和结果保存。
7. 下单幂等处理。
8. 日志和备份。

### 19.2 前端测试

使用 Playwright 做核心流程 E2E。

必须覆盖：

1. 打开首页。
2. 切换六大菜单。
3. 保存系统设置。
4. 运行环境检测。
5. 查看数据中心表格。
6. 新建 Python 策略。
7. 保存并检查策略。
8. 运行策略生成信号。
9. 创建回测任务。
10. 信号下单弹窗确认。

### 19.3 QMT Mock 模式

第一版开发必须提供 Mock 模式。

原因：

1. Codex 开发环境可能没有 QMT。
2. 前端页面需要假数据调试。
3. 自动测试不能依赖真实交易环境。

Mock 模式要求：

1. 模拟连接状态。
2. 模拟账户资金。
3. 模拟持仓。
4. 模拟委托成交。
5. 模拟行情数据。
6. 模拟下单返回。
7. 明确显示当前为 Mock，不是真实交易。

### 19.4 总体验收

第一版通过验收的标准：

1. 一键启动成功。
2. 六大菜单可访问。
3. UI 专业、清晰、无明显错位。
4. 数据中心可同步或 Mock 同步数据。
5. 策略开发可新建、编辑、检查、运行 Python 策略。
6. 策略信号可保存和展示。
7. 回测研究可创建任务并展示结果。
8. 交易执行可手动下单、信号下单、撤单，至少 Mock 模式可完整跑通。
9. 系统管理可环境检测、查看日志、备份恢复。
10. 所有关键错误都有中文提示和技术详情。
11. 所有长任务都有任务状态和进度。
12. 列表分页、筛选、刷新可用。

```text
系统启动
系统设置
环境检测
数据同步
数据展示
Python策略编辑
Python策略运行
策略信号展示
单策略回测
手动下单
信号下单
委托成交同步
日志备份
```

不属于以上链路的功能，默认不开发。

### 23.2 需求变更原则

新增需求必须满足至少一条：

1. 会影响主链路跑通。
2. 会影响交易安全。
3. 会影响数据正确性。
4. 会影响策略运行或回测可信度。
5. 会影响页面可操作性。
6. 会影响性能和稳定性。

如果只是“看起来更高级”，第一版不做。

### 23.3 模块优先级

| 优先级 | 模块/能力 | 原因 |
|---|---|---|
| P0 | 系统启动、环境检测、数据同步、策略运行、交易执行 | 没有这些主链路跑不通 |
| P1 | 回测、数据质量、日志、备份、运行监控 | 影响可信度和排错 |
| P2 | 图表优化、导出、历史版本对比 | 提升体验 |
| P3 | 多数据源、多账户、多策略组合、参数寻优 | 未来商业化再做 |

## 26. 策略运行安全与隔离规范

### 26.1 基本原则

用户策略是本地 Python 代码。第一版不承诺完全沙箱化，但必须做最小隔离，避免策略误操作交易和系统资源。

### 26.2 必须隔离

1. 策略拿不到 QMT 原始连接。
2. 策略拿不到交易下单服务。
3. 策略拿不到数据库连接对象。
4. 策略通过 `StrategyContext` 读数据。
5. 策略只能返回信号。
6. 策略运行有超时。
7. 策略运行日志被捕获。
8. 策略运行失败不影响主服务。

### 26.3 建议限制

1. 策略在子进程中运行。
2. 策略工作目录限制在项目目录内。
3. 不把系统配置和账户敏感信息传给策略。
4. 对策略输出做严格校验。
5. 策略信号金额超过系统最大单笔金额时，交易执行模块再次确认或拦截。

### 26.4 明确风险提示

由于 Python 本身可执行任意本地代码，用户从 AI 或外部复制策略代码前，应在代码编辑器中检查。系统可以提供“策略代码安全提示”，但第一版不做完整恶意代码检测。

## 28. 交易执行深度规范

### 28.1 下单流程

```text
前端点击下单
  ↓
前端弹窗确认
  ↓
前端按钮锁定
  ↓
后端生成 idempotency_key
  ↓
创建本地订单：待提交
  ↓
调用 QMT 下单
  ↓
保存 qmt_order_id
  ↓
更新状态：已提交/失败
  ↓
前端提示结果
  ↓
后台同步委托成交
```

### 28.2 信号下单流程

```text
读取未处理信号
  ↓
用户点击下单
  ↓
检查信号是否已处理
  ↓
生成本地订单
  ↓
关联 signal_id 和 strategy_id
  ↓
提交 QMT
  ↓
更新信号状态为已下单
```

同一信号只能生成一个有效订单。

### 28.3 撤单流程

```text
用户点击撤单
  ↓
确认弹窗
  ↓
检查订单状态是否可撤
  ↓
调用 QMT 撤单
  ↓
更新本地状态
  ↓
同步最新委托状态
```

### 28.4 订单状态映射

QMT 原始状态必须映射到内部状态。

内部状态固定：

```text
待提交
已提交
已报
部分成交
全部成交
已撤
废单
失败
未知
```

状态映射必须集中在：

```text
order_status_mapper.py
```

禁止在前端或多个 service 里重复写映射逻辑。

### 28.5 幂等规则

幂等键组成建议：

```text
account_id + symbol + side + price + quantity + source + signal_id + 时间窗口
```

第一版至少做到：

1. 前端按钮锁定。
2. 后端检查 idempotency_key。
3. 同一 signal_id 不允许重复生成有效订单。

## 29. 数据质量与数据一致性规范

### 29.1 数据标准化

所有 QMT 返回数据必须经过标准化。

标准化内容：

1. 股票代码格式。
2. 时间格式。
3. 数字类型。
4. 订单方向。
5. 订单状态。
6. 空值处理。

### 29.2 数据唯一性

必须通过唯一索引防止重复写入。

| 数据 | 唯一键 |
|---|---|
| 股票基础 | symbol |
| 日 K | symbol + trade_date |
| 分钟 K | symbol + period + datetime |
| 委托 | local_order_id 或 qmt_order_id |
| 成交 | trade_id 或 qmt_trade_id |
| 信号 | strategy_id + symbol + action + signal_time，可按实际调整 |

### 29.3 数据同步一致性

1. 同步成功后更新游标。
2. 同步失败不更新游标。
3. 部分成功要记录成功数量和失败数量。
4. 重试时从游标或指定区间开始。
5. 手动全量同步必须二次确认。
